# HARDWARE_TEST_AI_AUX_KO

## 2026-04-22 MCP2515 INT 판정 업데이트
- 확정: MCP2515 `INT_N` 하드웨어와 Portenta `D11/PH8` 입력은 사용 가능하다.
- 확정: `D11` 단독 입력 읽기와 `attachInterrupt(D11, FALLING)` 최소 진단은 정상 동작했다.
- 확정: 실패했던 것은 MCP2515 `INT_N`을 single falling-edge gate처럼 사용한 이전 본 펌웨어 구조다. 이 실패를 하드웨어 불가로 해석하지 않는다.
- 최종 적용: `portenta_h7_m7_mcp_int_main`은 `INT_N`을 active-low level hint로만 사용하고, 실제 RX 권한은 MCP2515 status/register drain state machine이 가진다. 2 ms bounded polling fallback을 유지한다.
- HIL 확인: PCAN 500 kbps, ID `0x202`, DLC 8, 10 ms 송신에서 `flags=0x0A`, `can_ok=1`, `can_drop=0`, `fault=0x00000000` 확인.
- 운영 규칙: MCP2515 INT 문제를 볼 때는 edge ISR 성공/실패만으로 판정하지 말고, `CANINTF`, `EFLG`, RX FIFO drain, fallback polling 유무를 함께 본다.

이 문서는 하드웨어 연결, 셋업, 시험, 검증 중에만 참고하는 보조 지침이다.
일반 코드 작업 규칙, shared 계약, board AGENTS 지침을 대체하지 않는다.

목적은 AI가 이미 실패한 추론, 이미 반증된 가설, 이미 검증 끝난 항목을 근거 없이 반복하지 않게 하는 것이다.

## 적용 범위
- 적용: 배선, 전원, 전압 레벨, 핀맵, 포트 인식, 펌웨어 업로드, 실측 로그, CAN/encoder/IO bring-up.
- 비적용: 일반 코드 리팩터링, 문서 정리, Qt 구현, shared record 계약 설계.
- 충돌 시 우선순위는 기존 `board/AGENTS.md`, `board/BRIEF.md`, `shared/docs/*`, 현재 소스 코드가 먼저다. 이 문서는 하드웨어 시험 중 판단 보조용이다.

## 시험 전 필수 구분
하드웨어 시험 답변이나 다음 조치를 내기 전에 아래 네 가지를 먼저 구분한다.

1. 확인된 사실
   - 로그, 전압, 핀맵, 포트, 빌드 결과, 사용자가 확정한 배선, 실제 수신 프레임처럼 증거가 있는 항목.
2. 아직 미확인 가설
   - 추정 원인, 모듈 내부 회로 추정, 노이즈 가능성, 부품 불량 가능성.
3. 이미 반증된 가설
   - 동일 조건에서 실패/성공 로그로 배제된 항목.
4. 이번 시험에서 달라진 점
   - 배선 1개, 전원 조건 1개, 펌웨어 1개, PCAN 설정 1개처럼 이전 실패와 다른 변수를 명시한다.

이 네 가지가 정리되지 않으면 같은 시험을 반복하지 않는다.

## 반복 시험 금지 규칙
- 실패 원인을 설명하지 못한 상태에서 같은 배선, 같은 펌웨어, 같은 포트, 같은 PCAN 설정으로 재시험을 요구하지 않는다.
- 재시험을 할 때는 "이전에는 무엇이 실패했고, 지금은 무엇을 바꿨기 때문에 다른 결과가 나와야 하는지"를 먼저 말한다.
- 같은 항목을 다시 의심하려면 새 증거가 있어야 한다. 예: 전압 실측 변화, 포트 변화, 펌웨어 변경, 배선 변경, 로그 변화.
- 사용자가 이미 확정한 물리 사실을 근거 없이 되돌리지 않는다. 예: 프린팅으로 확인한 CS 위치, VCC/GND 방향, 통전 확인된 점퍼선.
- 진단 펌웨어와 메인 펌웨어를 혼동하지 않는다. 시험 전에 현재 보드에 올라간 펌웨어 종류를 명시한다.

## 증거 우선순위
판단은 아래 순서로 한다.

1. 실제 로그: typed stream, `can_ok`, `can_drop`, raw register, upload 결과, COM 포트.
2. 실측값: 전원, GND 기준 전압, free wire와 connected node 비교, 종단저항.
3. 핀맵/코드: `include/BoardPins.h`, `platformio.ini`, 현재 빌드 플래그.
4. 사진/실크/데이터시트: 방향과 핀명 확인.
5. 추정: 모듈 내부 회로, 불량 가능성, 노이즈 가능성.

추정은 증거처럼 말하지 않는다. 추정으로 조치를 지시할 때는 어떤 측정으로 확인할지 함께 제시한다.

## 이번 MCP2515 bring-up에서 확정된 기준
아래 항목은 같은 조건에서 다시 원점부터 의심하지 않는다.

- Portenta `D8` COPI와 `D10` CIPO의 SPI 물리 경로는 loopback으로 정상 확인됐다.
- Portenta `D7` CS 출력과 점퍼선은 free wire 상태에서 약 `0V <-> 3.17V` 토글이 확인됐다.
- MCP2515 raw register가 `0xFF` 또는 `0x00`으로 고정되는 상태는 CAN FIFO 문제가 아니라 SPI/register 접근 실패로 먼저 본다.
- `setBitrate()` 전후 초기화 실패는 PCAN 송신량, CAN FIFO, CAN 종단보다 SPI/CS/전원/클럭 조건을 먼저 본다.
- PCAN이 500 kbps로 ID `0x301`, DLC 8 데이터를 송신할 때 현재 보드는 `CAN_RX_RAW`를 수신했고, `can_ok=1`, `can_drop=0`가 확인됐다.
- 성공한 벤치 조건은 MCP2515+TJA1050 모듈 5 V 전원, 공통 GND, `CS/SCK/SI/SO` 레벨변환, `MCP_8MHZ`, `CAN_500KBPS`, INT 미사용 polling이다.
- 최초 실패에서 중요한 차이는 INT 유무 자체가 아니라, MCP2515 5 V 구동 상태에서 CS만 3.3 V 직결이었다는 점이다. 이후 CS를 레벨변환 대상에 포함하고 INT를 뺐을 때 성공했다.
- 4채널 레벨컨버터 1개만 있을 때는 `CS/SCK/SI/SO`가 우선이다. INT는 optional이며 polling 성공이 확인됐다.
- 최종 MCP2515 1개 구성에서 INT까지 쓰려면 총 5개 신호 `CS/SCK/SI/SO/INT`를 레벨변환한다. 4채널 레벨컨버터 2개를 쓰면 충분하다.

## 잘못된 추론 패턴과 방지 규칙
### 1. 현상을 너무 넓게 요약하지 않는다
잘못된 패턴: "3.3 V 직결 실패"처럼 실제 배선을 뭉뚱그려 말한다.

규칙: 실패 조건은 실제 연결 단위로 쓴다. 예: "MCP 전원은 5 V, SCK/SI/SO/INT는 레벨변환, CS만 3.3 V 직결".

### 2. 이미 검증된 핀맵을 반복 의심하지 않는다
잘못된 패턴: loopback과 CS free-wire 측정 후에도 핀 순서, D7 배선, COPI/CIPO 매핑을 계속 재의심한다.

규칙: 새 증거 없이 핀맵 의심으로 되돌아가지 않는다. 이후 실패는 connected node 부하, 레벨변환, 모듈 전원 조건처럼 아직 미확인인 쪽으로 좁힌다.

### 3. CAN 문제와 SPI 문제를 섞지 않는다
잘못된 패턴: MCP register 접근 실패를 CAN FIFO overflow나 PCAN/종단 문제로 설명한다.

규칙: raw register 접근이 실패하면 CAN bus 이전 단계다. 먼저 CS, SCK, SI, SO, 전원, GND, MCP crystal, SPI 속도를 본다.

### 4. 코드 보완과 원인 해결을 구분한다
잘못된 패턴: SPI 속도 저하나 polling 구조 변경을 전압 레벨 문제의 해결처럼 말한다.

규칙: 코드 변경은 "완화/안정화/진단성 개선"인지 "원인 제거"인지 구분한다. 이번 성공의 원인 제거는 CS 포함 SPI 4선 레벨변환과 5 V 모듈 조건이다. SPI 250 kHz는 레벨컨버터 조건의 안정화다.

### 5. INT와 CS의 역할을 혼동하지 않는다
잘못된 패턴: INT를 빼서 성공했는지, CS를 살려서 성공했는지 구분하지 않는다.

규칙: MCP2515 SPI에서 CS는 필수, INT는 선택이다. 4채널 제약에서는 INT를 빼고 CS를 레벨변환하는 판단이 우선이다. 부하가 커지는 최종안에서는 별도 채널로 INT를 살린다.

## 하드웨어 시험 답변 형식
시험 중 사용자에게 지시하기 전에는 가능하면 짧게 아래 형식을 따른다.

- 현재 확정: 이미 증거가 있는 사실.
- 미확인: 아직 가설인 항목.
- 이번 변경점: 이전 실패와 달라진 단일 변수.
- 다음 시험: 한 번에 한 목적만 검증하는 조치.
- 성공/실패 판정 기준: 어떤 로그나 전압이면 통과/탈락인지.

긴 절차보다 판정 기준을 우선한다. 사용자가 이미 작업 중이면 필요한 지시만 짧게 준다.

## 중단 기준
다음 상황에서는 같은 방향의 시험을 멈추고 원인 분류를 다시 한다.

- 같은 raw register 값 `0xFF` 또는 `0x00`이 같은 조건에서 반복된다.
- 전압 측정이 free wire와 connected node에서 다르게 나온다.
- 업로드된 펌웨어가 진단용인지 메인인지 불명확하다.
- 사용자가 이미 같은 배선을 반복 확인했다고 말한다.
- 새 하드웨어 조건 없이 같은 PCAN/종단/포트 확인만 반복하게 된다.
